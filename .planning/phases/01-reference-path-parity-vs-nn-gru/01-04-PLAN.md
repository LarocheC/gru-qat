<objective>
Add the fourth and final parametrized parity-test family to `tests/test_layer_parity.py`: random initial hidden state parity. Two new functions — fast over `FAST_GRID`, slow over `SLOW_GRID` — comparing `(out, h_T)` between `GRULayer` and `nn.GRU` when `h_0` is a random tensor (not zeros).

Purpose: REF-02 requires the parity contract to hold with non-zero initial state. A zero-h0 implementation can "accidentally" work if the time-loop initialization is wrong (e.g., reading from `h0=None` separately from the loop body); the random-h0 variant catches that. Per CONTEXT specifics, this family asserts BOTH `out` and `h_T` in one test — it is the h_0≠0 isolation family, not a fwd-vs-h_T family-split.

Output: 2 new parametrized test functions appended to the test file.

    Function 1 — `test_layer_with_random_h0_matches_nn_gru(T: int, B: int, H: int) -> None`. Decorator: `@pytest.mark.parametrize("T,B,H", FAST_GRID)`. Body:

    Step 1 (setup): `torch.manual_seed(0)`, `IN = max(H, 1)`, `layer = _make_dense_fp32_layer(IN, H)`, `gru = _translate_cell_to_nn_gru(layer)`.

    Step 2 (random h0, shape-consistent for both implementations): `x = torch.randn(T, B, IN)`. `h0_3d = torch.randn(1, B, H)` (nn.GRU shape). `h0_2d = h0_3d.squeeze(0)` (GRULayer shape — same underlying values, just stripped leading axis). NOTE: do NOT clone — `squeeze(0)` returns a view with shared storage, which is fine because neither call modifies h0 in-place (this test is forward-only; no requires_grad needed).

    Step 3 (forward both paths): `out_ref, hT_ref = gru(x, h0_3d)`. `out_ours, hT_ours = layer(x, h0_2d)`.

    Step 4 (per-name assertion loop — both out and h_T in one test): build a list `[("out", out_ref, out_ours), ("h_T", hT_ref.squeeze(0), hT_ours)]` and loop with the relative-error idiom: `rel = (ref_t - our_t).abs().max().item() / max(ref_t.abs().max().item(), 1e-6); assert rel < 1e-4, f"{name} rel diff {rel:.4e} (T={T},B={B},H={H},h0=rand)"`. The `h0=rand` suffix in the failure message distinguishes this family from the zero-h0 fwd / h_T families.

    Docstring: "h_0 != 0 parity over the fast grid. Asserts both `out` and `h_T` in the same test — this is the h_0 != 0 isolation, not a fwd-vs-h_T family split (D-09 + CONTEXT Specifics). Catches initialization bugs where a path special-cases h_0=None but mishandles a real tensor."

    Function 2 — `test_layer_with_random_h0_matches_nn_gru_slow(T: int, B: int, H: int) -> None`. Identical body. Decorators: `@pytest.mark.slow` above `@pytest.mark.parametrize("T,B,H", SLOW_GRID)`. Docstring mentions the slow grid.

    Implementation notes:
    - The seed `torch.manual_seed(0)` is reset at the top of each test body so `h0_3d` and `x` are deterministic regardless of test ordering (TESTING.md "Bench-Style Smoke Tests vs. Correctness Tests").
    - There is NO autograd in this test family — no `requires_grad`, no `.backward()`. h_0!=0 backward is implicitly covered by Plan 03's `test_layer_backward_matches_nn_gru` because that test passes `h0_ref` / `h0_ours` as requires_grad leaves AND uses `torch.zeros` for the values. If a grad-on-random-h0 issue surfaced, the gradient test would have caught it via `dh_0` — no need to duplicate here.
    - The `h_T` shape contract: nn.GRU returns `h_n` as `[1, B, H]`; GRULayer returns `h_T` as `[B, H]`. The squeeze on the ref side matches Plan 02's pattern.

    Do NOT add `xfail`. Do NOT fuse with the forward/h_T families. Do NOT touch the helpers, constants, or other tests. The single test does both assertions because the SPEC for the h_0!=0 family says both are checked together — splitting it would create 8 grid families instead of 4, which D-09 explicitly rejects.
