<!-- bd workflow contract (from CLAUDE.md "Beads Issue Tracker" + project's bd remember). -->

bd CLI commands used in this plan:
- `bd create --title "<test_function_name>[<parametrize_id>]"` — create a new finding.
- `bd update <id> --claim` — claim it for the in-progress fix.
- `bd update <id> --notes "<text>"` — attach the pytest tail or fix-commit sha.
- `bd close <id>` — mark fixed after Commit B is in.
- `bd list` — verify the audit-finding inventory.
- `bd ready` — confirm no unresolved findings remain at phase exit.

Failure-message anatomy from Plans 02–04:
- Forward family: `out rel diff <X> (T=<T>,B=<B>,H=<H>)`
- h_T family: `h_T rel diff <X> (T=<T>,B=<B>,H=<H>)`
- Backward family: `<grad_name> rel diff <X> (T=<T>,B=<B>,H=<H>)` where `grad_name` ∈ {dx, dh_0, dW_ih, dW_hh, db_ih, db_hh}.
- h_0!=0 family: `<name> rel diff <X> (T=<T>,B=<B>,H=<H>,h0=rand)` where `name` ∈ {out, h_T}.

Two-commit discipline (D-10):
- Commit A — failing test, no src/ changes. (The tests are already in place from Plans 01–04 — Commit A in the sense of "first commit where the test fails on the codebase as-of-now" is the most recent commit on this branch BEFORE any fix lands.)
- Commit B — fix in src/, no test changes. Commit message references the bd id.

Anti-pattern (D-12): never `@pytest.mark.xfail`. The test stays as a hard failure until Commit B fixes it.

Locked tolerance (PROJECT.md Constraints + ROADMAP success criterion 4):
- `tests/test_parity.py` cell-level `< 1e-5` parity gate is locked. Don't loosen.
- Phase 1 layer-level tolerance is `< 1e-4` against nn.GRU; if a finding shows ~1e-4 noise at very large T due to accumulated arithmetic drift (not a math bug), discuss before any fix — the conservative fix is to mark that specific parametrize id `slow` rather than touch tolerance. Log the decision in `bd remember` for cross-session continuity, NOT in PROJECT.md (PROJECT.md changes need user approval — that's why this plan is non-autonomous).
